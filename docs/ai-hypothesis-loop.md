# Manual AI Hypothesis Loop

**Repository:** `cellauto-lab`
**Status:** Design — v2 planning
**Date:** 2026-06-18

---

## Purpose

The Manual AI Hypothesis Loop is the next phase of the `cellauto-lab` workflow. Its purpose is to turn the structured experiment summaries generated by v1 into falsifiable scientific hypotheses that can be tested with follow-up simulations.

In v1, the notebook simulates all 256 elementary cellular automaton (ECA) rules, computes seven behavioral metrics, and produces structured summaries. Those summaries contain explicit placeholder fields (`interpretation_placeholder`, `hypothesis_placeholder`, `next_experiments_placeholder`) that v1 leaves empty. The AI hypothesis loop defines how those fields get filled — through human-mediated interaction with an LLM — and how the resulting hypotheses are evaluated and converted into reproducible experiments.

The goal is not automation. The goal is to establish a governed, auditable process that separates what the simulation measured from what an LLM infers, and to ensure that every inference is testable.

---

## Why v2 Is Manual, Not Automated

Automating LLM calls inside the notebook would be faster but introduces risks that are not acceptable at the current stage of the project:

| Risk | Why It Matters |
|---|---|
| Hallucinated citations | An LLM under time pressure may invent references to Wolfram or other literature. Automated pipelines would propagate these unchecked. |
| Unjustified causal claims | A single simulation run on a 100-cell, 200-step grid is not sufficient evidence for most causal claims. Automation removes the human checkpoint that catches these. |
| Schema drift | If the LLM response format changes between API versions, an automated pipeline may silently accept malformed output. |
| Governance traceability | SYNTRAN AIEOS requires that LLM contributions be explicitly attributed and reviewed. A manual step makes attribution clear. |
| Trust calibration | The project is in incubation. The value of LLM hypothesis generation has not yet been validated against human judgment. Manual review is the calibration step. |

The manual approach is not a limitation — it is the governance design. Automation can be added in v3 once the quality of LLM hypotheses has been validated against a baseline set of manually reviewed examples.

## v1 Inputs Available from the Notebook

When the notebook is run with `Restart & Run All`, it produces:

### `ai_hypothesis_loop_inputs`

A list of structured summary dictionaries for the top-ranked rules. Each summary has the following format (from `cellauto_lab.reporting.experiment_summary()`):

```json
{
  "experiment_id": "rule_030_single_seed",
  "timestamp_utc": "2026-06-18T12:00:00Z",
  "rule_number": 30,
  "parameters": {
    "n_cells": 100,
    "n_steps": 200,
    "initial_condition": "single_seed",
    "random_seed": null,
    "boundary_condition": "fixed_zero"
  },
  "observation": "Rule 30 simulated for 200 steps over 100 cells. Initial condition: single_seed. Boundary condition: fixed_zero.",
  "metrics": {
    "density_mean": 0.501,
    "density_final": 0.49,
    "transition_count": 24.8,
    "activity_score": 0.499,
    "entropy_score": 3.29,
    "periodicity_score": 0.0,
    "compression_ratio": 0.89
  },
  "interpretation_placeholder": "",
  "hypothesis_placeholder": "",
  "next_experiments_placeholder": []
}
```

The `observation` field is factual only. The three `_placeholder` fields are empty. The LLM's job is to provide content for those placeholders — but in the v2 loop, that content is returned in a separate structured response document, not written back into the summary directly.

### Serialized JSON for LLM submission

The notebook serializes `ai_hypothesis_loop_inputs` using `cellauto_lab.reporting.summaries_to_json()`, producing a JSON string that can be pasted directly into a prompt.

## Loop Stages

### Stage 1 — Run Simulations

Configure and run the notebook with `Restart & Run All`. This simulates all 256 rules under the v1 default parameters:

- 100 cells, 200 steps
- `single_seed` initial condition
- `fixed_zero` boundary condition
- Random seed 42 (for any random IC runs)

Record the notebook execution timestamp.

### Stage 2 — Collect Metrics

The notebook computes all seven metrics for every rule and assembles a ranked DataFrame. Rules are ranked by a heuristic interest score that combines high activity, high entropy, low periodicity, and intermediate density.

Verify that the metrics are reproducible by comparing the output against a previously recorded reference run (if one exists).

### Stage 3 — Generate Structured Summaries

The notebook calls `experiment_summary()` for the top-ranked rules and serializes the results using `summaries_to_json()`. The output is stored in `ai_hypothesis_loop_inputs`.

Verify that:
- All summaries have the correct key structure
- All placeholder fields are empty strings or empty lists
- All `observation` fields are purely factual

### Stage 4 — Submit Summaries to LLM

Copy the JSON output from `ai_hypothesis_loop_inputs` and use it as input to the prompt template in `prompts/hypothesis-review.md`.

Fill in the template placeholders:
- `{{RESEARCH_QUESTION}}`: paste from `docs/research-question.md`
- `{{EXPERIMENT_SUMMARIES_JSON}}`: paste the JSON from `summaries_to_json()`
- `{{KNOWN_LIMITATIONS}}`: paste the limitations section from `docs/methodology.md`

Submit the completed prompt to an LLM session (e.g., Claude via claude.ai). Do not use an API call at this stage.

### Stage 5 — Receive Falsifiable Hypotheses

The LLM should return a structured JSON response that conforms to `schemas/hypothesis_response.schema.json`. Each hypothesis must include:

- The rule(s) it applies to
- The specific metric(s) it references
- A falsifiable prediction
- A concrete test plan
- A stated confidence level and risk of overclaiming

If the LLM returns free-form prose instead of structured JSON, discard the response and re-submit with a stronger instruction to follow the schema. Do not manually convert prose to JSON — the structure is the evidence of discipline.

### Stage 6 — Review Hypotheses for Quality

Apply the checklist in `docs/hypothesis-quality-checklist.md` to each hypothesis. Assign each one of the following statuses:

| Status | Meaning |
|---|---|
| `proposed` | Default — not yet reviewed |
| `accepted_for_testing` | Passes quality checklist; has a concrete test plan |
| `rejected` | Fails checklist (vague, unfalsifiable, hallucinated citation, overclaiming) |
| `inconclusive` | Interesting but cannot be tested with current v1 tooling |

Document the rejection reasons. These are part of the audit trail.

### Stage 7 — Convert Accepted Hypotheses to Follow-Up Experiments

For each `accepted_for_testing` hypothesis, define a follow-up experiment by specifying:

- Rule(s) to simulate
- Parameter changes from v1 defaults (e.g., different initial condition, grid size, step count)
- Metrics to compare against the v1 baseline
- Expected direction of change (e.g., "periodicity_score should increase under periodic boundary condition")

Record the experiment specification in a structured format before running it. This prevents post-hoc rationalization of results.

### Stage 8 — Compare Expected vs Observed Results

Run the follow-up simulations. Compare the observed metric values against the predictions from Stage 7.

For each hypothesis, record:
- Whether the prediction was confirmed, refuted, or inconclusive
- The actual metric values observed
- Whether the parameter change produced the expected behavioral change
- Any anomalies or unexpected behavior worth flagging

Update the hypothesis status accordingly. Hypotheses that survive repeated testing become candidates for formal documentation in `docs/references.md`.

## What Counts as a Good Hypothesis

A good hypothesis in this context:

1. **Is falsifiable** — there is a specific simulation that could disprove it
2. **Names specific rules** — "Rule 30" not "chaotic rules in general"
3. **Names specific metrics** — "periodicity_score" not "regularity"
4. **Predicts a direction** — "increasing n_cells should decrease compression_ratio" not "compression_ratio will change"
5. **Acknowledges the evidence base** — states which metric values from which run it is based on
6. **States confidence honestly** — "low" if based on a single run with one initial condition
7. **Proposes a concrete test** — specifies the parameter change and the expected outcome
8. **Does not cite external literature** — unless the citation is to a document in this repository

**Example of a good hypothesis:**

> Rule 30 shows a compression_ratio of 0.89 and periodicity_score of 0.0 under single_seed initial conditions with n_cells=100 and n_steps=200. If this reflects genuine pseudo-random behavior rather than transient instability, then increasing n_cells to 500 should not significantly decrease compression_ratio (expected: compression_ratio remains above 0.80). Confidence: low. Risk of overclaiming: medium — finite-size effects and boundary artifacts may dominate at n_cells=100.

---

## What Counts as a Bad Hypothesis

Reject any hypothesis that:

1. **Is not falsifiable** — "Rule 30 is interesting" cannot be tested
2. **Makes causal claims from a single run** — "Rule 30 generates true randomness" is a mathematical claim, not a simulation result
3. **Cites invented references** — any citation not present in `docs/references.md` is suspect
4. **Is vague about metrics** — "high complexity" without naming a metric is not testable
5. **Does not propose a concrete test** — "more research is needed" is not a test plan
6. **Overstates confidence** — "high" confidence on a single 100-cell run is unjustified
7. **Mixes observation and interpretation in the same sentence** — "Rule 30 produces chaotic output (entropy=3.29)" conflates a measurement with a label

**Example of a bad hypothesis:**

> Rule 30 is Wolfram's Class IV rule and produces true randomness, as confirmed by Wolfram (2002). Its high entropy proves it is maximally complex.

This fails because: (1) Rule 30 is Class III, not Class IV; (2) "true randomness" is a mathematical claim not supported by simulation evidence; (3) the citation cannot be verified in this repository; (4) entropy does not "prove" complexity.

## Examples Using Elementary Cellular Automata

### Example 1 — Testable hypothesis from Rule 90

**Observation basis:** Rule 90 produces a Sierpinski-triangle pattern under single_seed initial conditions. Its density_mean is approximately 0.5 and periodicity_score is 0.0 under the v1 default parameters.

**Hypothesis:** Rule 90's density_mean under random initial conditions with n_cells=100 and n_steps=200 will converge to a value close to the single_seed result (within ±0.05), because Rule 90 is an additive rule whose long-run density is determined by the rule rather than the initial condition.

**Test plan:** Run `simulate(90, n_cells=100, n_steps=200, initial_condition='random', random_seed=42)` and compare `density_mean` against the single_seed result.

**Status:** `accepted_for_testing`

### Example 2 — Hypothesis that should be rejected

**Claim:** "Rule 110 is Turing-complete, therefore its compression_ratio should approach 1.0."

**Rejection reason:** Turing completeness is a mathematical property that cannot be measured or falsified by a 200-step simulation on a finite grid. The compression_ratio may correlate with complexity, but it does not measure computational universality. This hypothesis conflates a mathematical theorem with a simulation metric.

## Limitations

- The v2 loop is limited by the v1 metric set — seven metrics that are proxies, not formal measurements of complexity
- Single initial condition (single_seed) results may not generalize across initial conditions
- The `fixed_zero` boundary condition may confound behavioral characterization for rules sensitive to boundary state
- LLM hypothesis quality depends on the prompt design and the model used; this has not been systematically evaluated
- The manual process creates a bottleneck — only a small number of rules can be reviewed per session
- There is no automated validation that a hypothesis was correctly converted into a follow-up experiment

## Future Automation Path

Once the manual loop has produced a validated baseline set of hypotheses (suggested threshold: at least 10 accepted hypotheses with confirmed or refuted outcomes), the following automation steps become credible:

1. **Schema validation at intake** — add a `jsonschema` validation step to the notebook that verifies LLM responses before recording them
2. **Automated experiment generation** — a v3 module that reads an accepted hypothesis and generates a `simulate()` call with the specified parameter changes
3. **API integration** — a governed v3 pipeline that submits summaries to a Claude API call inside the notebook, subject to SYNTRAN AIEOS approval
4. **Hypothesis tracking** — a structured log file (`hypotheses/`) recording all proposed, accepted, rejected, and tested hypotheses across sessions
5. **Comparative analysis** — automated comparison of predicted vs observed metric values, stored as a CSV for longitudinal analysis

None of these steps should be implemented before the manual loop has produced validated outputs.
